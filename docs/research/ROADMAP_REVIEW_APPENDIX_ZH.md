# Perilla 行业化能力路线图 — 评审附录

更新时间：2026-05-27

## 文档说明

本附录对以下三条能力演进方向进行评审，供规划、立项与资源排期参考：

1. **完善行业级场景化适配支撑**
2. **强化企业级的安全治理能力**
3. **提升行业级可观测性支撑能力**

评审基线：Perilla 仓库当前实现与文档（`ARCHITECTURE.md`、`AGENTS.md`、`docs/workflow/`、`docs/ops/SAAS_PUBLIC_LAUNCH_GATE_ZH.md` 等）。

## 一、总体判断

三条主线（**场景化适配 → 企业安全治理 → 可观测与合规审计**）与 Perilla「本地/专网 + 网关中心化 + 工作流可审计」定位**高度一致**，尤其适合**监管、科研**场景。作为**行业化/企业化路线图**方向正确且必要。

需认清：Perilla 当前更接近**能力完备度较高的 AI 中台内核**（推理网关、Agent、Workflow、租户、基础审计、Helm 部署、Prometheus 埋点）；三条描述的是**行业解决方案层（Solution Layer）**，中间尚有「产品化、认证化、生态对接」一整层工作。若不加阶段切分，易与「Local-first、User-in-Control、Determinism over Magic」原则产生张力（如行业协议全家桶、第三方合规 SaaS 深度绑定）。

**建议定位**：将三条作为 **Perilla Enterprise / Gov-Research Edition** 的顶层能力域；**开源核心**保持可插拔，**行业包**可独立交付与版本节奏。

## 二、分项评审

### 2.1 完善行业级场景化适配支撑

**规划要点（原文归纳）**

- 重点增加监管、科研类场景常用**行业级协议、标准**的适配支持
- 提供与**行业典型周边生态系统**的适配对接能力
- 开发**场景化合规性治理工具包**，覆盖典型合规治理需求
- 提供**场景化参考架构、部署示例和合规性验证工程模板**，便于快速落地

#### 价值与必要性：高

监管、科研采购与验收语言通常是：**标准接口、周边系统、合规工具包、参考架构**——而非仅「能跑 Workflow」。上述四类对应 **PoC → 可验收** 的四道关。

#### 与现状契合点

| 诉求 | 现状（可借力） | 缺口 |
|------|----------------|------|
| 场景化编排 | Fork/Join、approval、checkpoint、verify_loop、Demo bundle | 缺行业命名的成套方案（监管报送、伦理审查、公文发布等） |
| 合规叙事 | AGENTS.md、治理审计、租户隔离、SAAS 上线门禁文档 | 缺**可签字**的行业合规模板与验证工程 |
| 生态 | MCP/Skill、HTTP 节点、API 开放 | 缺 OA、档案、科研管理、监管数据平台的**认证级连接器** |

#### 风险与注意

1. **「行业级协议、标准」范围易失控**  
   监管侧可能涉及电子公文、档案、数据共享交换；科研侧涉及伦理审查、项目库、文献元数据等。不宜一口一个「协议适配层」，应先选 **2～3 个可复用、可验收** 的接口形态（统一审批回调、结构化报送 JSON Schema、知识库元数据交换），其余用 **HTTP/Webhook + 映射模板** 覆盖。

2. **合规工具包须对齐 User-in-Control**  
   工具包应为**显式策略**（审批必选、字段校验、敏感词/分级标签检查节点），避免黑盒「自动合规」——这可强化 Perilla 差异化。

3. **参考架构不宜铺得过宽**  
   建议：**1 套监管专网参考架构 + 1 套科研院所参考架构 + 合规验证工程（自动化检查清单）**，避免 N 个行业克隆导致维护不可持续。

#### 评审结论

| 维度 | 结论 |
|------|------|
| 战略方向 | 正确 |
| 建议优先级 | **P0**：场景包（参考架构 + 验证工程 + Playbook/Demo）；**P1**：精选协议与连接器 |
| 与核心关系 | 行业包宜 Enterprise 交付，核心保持通用节点与网关 |

### 2.2 强化企业级的安全治理能力

**规划要点（原文归纳）**

- 与企业级**集中式密钥管理、身份认证**类系统适配
- **流量加密、数据加密、签名验签**完整支撑
- **高可用集群部署**直接支持
- 更多**可插拔安全治理组件**，对接第三方安全治理服务
- **覆盖行业级合规性标准的硬性指标要求**

#### 价值与必要性：很高（常为政企单硬门槛）

从「能部署」到「能过评审」的分水岭，与监管/科研专网部署强相关。

#### 与现状对照

| 能力项 | 现状（概括） | 评价 |
|--------|----------------|------|
| 身份认证 | API Key + RBAC；`X-Tenant-Id`；非完整 IdP 联邦 | **需补强**：OIDC/SAML/LDAP 与企业 IAM |
| 密钥管理 | 环境变量/Secret、模型 Key 元数据 | **需补强**：KMS/Vault、轮换与审计 |
| 传输/存储加密 | HTTPS、Helm 门禁、`SECURITY_GUARDRAILS_STRICT` | **部分具备**；国密、字段级加密、验签需单独立项 |
| 高可用 | Helm、`docker-compose.prod` | **有基础**；缺官方多副本、队列/DB HA **一键基线** |
| 可插拔安全组件 | Plugin、中间件链 | **架构可扩展**；需 Security Plugin 契约（脱敏、DLP、审计 sink） |
| 硬性合规指标 | `SAAS_PUBLIC_LAUNCH_GATE_ZH.md`、启动护栏 | **工程门禁有**；≠ 等保/密评/关保条目自动满足 |

#### 风险与注意

1. **「覆盖行业硬性指标」表述宜收敛**  
   等保 2.0、密评、数据安全法、生成式 AI 备案等，**无单一产品可全覆盖**。可行路径：**控制项映射表 + 配置基线 + 审计证据导出**，文档声明边界。

2. **与 Local-first 协调**  
   第三方 DLP/UEBA 等多需出站；设计为**可选旁路**，默认关闭，专网用本地实现。

3. **HA 须定义 SLA 档位**  
   区分 **Standard HA**（三副本 + Redis + PG）与 **Advanced HA**（双活/跨 AZ），避免客户预期错位。

#### 评审结论

| 维度 | 结论 |
|------|------|
| 战略方向 | 正确，**资源投入优先级最高** |
| 建议优先级 | **P0**：IAM 联邦 + KMS/Vault + HA 基线；**P1**：国密/验签、Security Plugin SDK |
| 与核心关系 | 牵动配置、部署、中间件顺序，需架构评审与版本说明 |

### 2.3 提升行业级可观测性支撑能力

**规划要点（原文归纳）**

- 与企业级**运维监控类系统**适配：采集、适配、配置、告警规则
- **场景化合规性审计报告**，支持符合行业要求的报告输出
- 增强**工作流执行追踪、复盘**，优化可观测性

#### 价值与必要性：高（ROI 常优于从零造场景）

监管、科研均要求：**谁、何时、对何数据、何决策、结果是否可复盘**。Perilla 已有 Workflow Timeline、审批 API、治理审计、governance snapshot、Prometheus、`/api/v1/audit` 等，本条线是**产品化既有能力、对外系统友好**，与编排卖点协同强。

#### 与现状对照

| 诉求 | 现状 | 缺口 |
|------|------|------|
| 企业监控对接 | Prometheus instrumentator | OTel 标准导出、告警规则包、Zabbix/蓝鲸等连接器 |
| 合规审计报告 | audit logs、执行记录 | 按行业模板排版的一键报告（监管留痕、科研伦理） |
| 工作流追踪复盘 | Timeline、call-chain；Agent 侧事件流/replay | 跨 Agent+Workflow 统一 correlation_id 与导出格式 |

#### 风险与注意

1. **「符合行业标准报告」**  
   多数为标准**单位制度 + 上级模板**；宜做**可配置报告模板 + 证据链导出（JSON/PDF）**，附 2～3 示例模板。

2. **告警规则避免泛滥**  
   提供**场景化规则包**（队列积压、审批超时、checkpoint 失败率、租户配额），而非海量通用规则。

3. **追踪须与脱敏一致**  
   与 `sensitive_data_redaction` 等中间件策略对齐，可配置字段级脱敏。

#### 评审结论

| 维度 | 结论 |
|------|------|
| 战略方向 | 正确，与现有架构协同最好 |
| 建议优先级 | **P0**：OpenTelemetry + 审计报告 MVP + 工作流复盘导出；**P1**：监控平台连接器 |
| 与核心关系 | 可复用 Workflow/approval/checkpoint 已有数据面 |

## 三、能力域依赖关系

建议实施顺序理解如下（存在迭代闭环）：

```
场景化适配包 ──依赖──► 安全治理基线
        ▲                    │
        │                    ▼
        └──── 可观测/合规报告 ◄┘
```

- **安全治理**：政企订单前提，宜略早于「纯场景包装」。
- **可观测/审计报告**：场景包验收的**证据层**，与 approval/checkpoint 同步产品化。
- **场景化适配**：可先用参考架构 + Demo/Playbook 启动，再固化协议与连接器。

## 五、落地建议（Epic 级）

### 5.1 场景化适配（Enterprise / Gov-Research）

| 优先级 | 交付物 | 说明 |
|--------|--------|------|
| P0 | 监管专网参考架构（文档 + Helm values 示例） | 含租户、审批、审计、无外网默认 |
| P0 | 科研院所参考架构 | 含 KB、Verify Loop、模型分档 |
| P0 | 合规验证工程模板 | 脚本/检查清单，对齐 `merge-gate` 风格 |
| P0 | 2 个认证级连接器 MVP | 建议：OA 审批回调、知识库元数据同步 |
| P1 | 行业 Schema 注册中心 | 报送/伦理/公文等 JSON Schema 可配置校验 |
| P2 | 扩展协议适配 | 按客户项目增量，避免 preemptive 全栈 |

### 5.2 企业安全治理

| 优先级 | 交付物 | 说明 |
|--------|--------|------|
| P0 | OIDC/OAuth2 登录与 API 访问统一 | 与现有 API Key 并存、可切换 |
| P0 | Vault/KMS 密钥引用 | 模型 Key、Webhook Secret 等 |
| P0 | Helm HA 基线 | 多副本 + Redis 限流 + DB；文档化 RTO/RPO 假设 |
| P1 | Security Plugin SDK | 脱敏、DLP hook、审计 sink |
| P1 | 控制项映射表 | 等保/密评常用条 ↔ Perilla 配置/日志证据 |
| P2 | 国密 TLS / 签名验签 | 单独立项，面向特定行业要求 |

### 5.3 可观测性与合规审计

| 优先级 | 交付物 | 说明 |
|--------|--------|------|
| P0 | OpenTelemetry 导出 | 追踪推理、Workflow 节点、审批事件 |
| P0 | 证据链导出 API | execution_id → 时间线 + 审批 + checkpoint 结果 |
| P0 | 可配置合规报告模板 ×2 | 监管留痕示例、科研实验记录示例 |
| P1 | 场景化 Prometheus 告警规则包 | 与 Helm 一并发布 |
| P1 | 主流监控平台集成指南 | Prometheus 已有，补充对接样例 |
| P2 | 跨 Agent/Workflow 统一复盘 UI | correlation_id 贯穿 |

## 七、结论摘要

| 能力域 | 方向评价 | 建议投入排序 |
|--------|----------|----------------|
| 行业级场景化适配 | 战略正确；忌协议栈铺过大 | 第二梯队（与场景包 P0 并行） |
| 企业级安全治理 | 政企硬门槛；缺口最大 | **第一梯队** |
| 行业级可观测性 | 与现有能力协同最好 | **第一梯队**（与安全并行） |

当前最大差距不在「能否编排」，而在：**身份与密钥企业化、可签字合规证据、周边系统认证级对接、HA/监控开箱程度**。若资源有限，排序建议：

**企业安全治理 ≈ 可观测/审计报告 > 行业协议广度**；

场景化以**参考架构 + 工具包 + 模板**快速形成可销售/可研究材料，与本目录主报告及 Demo/Playbook 形成闭环。
